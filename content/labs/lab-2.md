## 2. Короткі теоретичні відомості

### 2.1 MySQL
Для цієї роботи обрано MySQL як реляційну систему керування базами даних. Вона ідеально підходить для сценарію сервісу бронювання, де є окремі таблиці зареєстрованих користувачів та їхніх записів на репетиції, пов'язані між собою зовнішніми ключами. Така структура гарантує цілісність даних (наприклад, видалення бронювань при видаленні користувача) і дозволяє легко масштабувати проєкт.

### 2.2 Node.js та Express
Серверну логіку реалізовано на платформі Node.js, а HTTP API побудовано за допомогою популярного фреймворку Express. Ця зв’язка забезпечує просту, надійну та гнучку маршрутизацію. У межах лабораторної роботи реалізовано RESTful API з маршрутами для реєстрації/авторизації користувачів та повним циклом управління бронюваннями (створення, отримання списку, редагування часу та скасування), що перевіряються як через клієнтський інтерфейс (Fetch API), так і через Postman.

### 2.3 Sequelize ORM
Sequelize використано як ORM-прошарок (Object-Relational Mapping) між JavaScript-кодом та MySQL. Замість ручного створення таблиць, структура бази даних описується через JavaScript-моделі (User, Booking), які автоматично синхронізуються з БД (sequelize.sync()). CRUD-операції виконуються через зручні вбудовані методи моделі на кшталт create, findAll, findByPk, update та destroy, що значно прискорює розробку та захищає від SQL-ін'єкцій.

### 2.4 Прямі SQL-запити
Окремо від ORM у проєкті створено демонстраційний скрипт (raw_sql_test.js) на базі пакета mysql2. Він призначений для виконання вимог лабораторної роботи щодо прямого виконання чистих SQL-запитів (SELECT, INSERT, UPDATE, DELETE) із середовища Node.js. Це наочно демонструє механіку роботи з базою даних на низькому рівні до використання абстракцій ORM.

## 3. Проєктування бази даних

### 3.1 Предметна область
У якості прикладної області взято сервіс оренди професійних репетиційних кімнат (DrumSpace). Система дозволяє музикантам реєструвати особистий кабінет, бронювати кімнати на визначену дату та час, переглядати історію своїх репетицій, а також переносити або скасовувати записи. Також передбачено функціонал "швидкого бронювання" для неавторизованих користувачів (гостей).

### 3.2 Основні сутності
У лабораторній версії проєкту використано дві основні сутності:

* User — зареєстрований користувач, який має унікальний email, ім'я та захешований пароль.
* Booking — запис на репетицію, що містить назву кімнати, дату та час, контактний телефон, дані гостя (якщо бронювання здійснено без авторизації) та зовнішній ключ для прив'язки до конкретного профілю користувача.

### 3.3 Зв’язки між таблицями
У моделі реалізовано класичний зв’язок типу One-to-Many (Один-до-багатьох):

* Один користувач може створити багато бронювань (``User.hasMany(Booking)``).
* Кожне конкретне бронювання належить лише одному користувачу (``Booking.belongsTo(User)``).

### 3.4 Схема даних (Sequelize Моделі)
Нижче наведено фрагменти коду, які описують структуру таблиць та зв'язки у проєкті:

#### Модель User (models/User.js):

```bash
const User = sequelize.define('User', {
    name: {
        type: DataTypes.STRING,
        allowNull: false
    },
    email: {
        type: DataTypes.STRING,
        allowNull: false,
        unique: true
    },
    password: {
        type: DataTypes.STRING,
        allowNull: false
    }
});
```

#### Модель Booking (models/Booking.js):

```bash
const Booking = sequelize.define('Booking', {
    roomName: {
        type: DataTypes.STRING,
        allowNull: false
    },
    content: {
        type: DataTypes.TEXT, // Зберігає дату та час
        allowNull: false
    },
    phone: {
        type: DataTypes.STRING,
        allowNull: false 
    },
    guestName: {
        type: DataTypes.STRING,
        allowNull: true 
    },
    guestEmail: {
        type: DataTypes.STRING,
        allowNull: true 
    }
});
```

#### Налаштування зв'язків (у server.js):

```bash
const User = require('./models/User');
const Booking = require('./models/Booking');

// Зв'язок One-to-Many
User.hasMany(Booking);
Booking.belongsTo(User);
```

## 4. Реалізація backend-частини

### 4.1 Структура серверної частини
Серверна частина розташована в кореневій папці проєкту та має класичну для Express-застосунків структуру. Основні файли розподілено так:

* `server.js` — головний файл: створення Express-застосунку, налаштування CORS, визначення маршрутів (API) та запуск сервера;
* `config/database.js` — ініціалізація підключення до бази даних через Sequelize;
* `models/User.js` — опис структури таблиці користувачів;
* `models/Booking.js` — опис структури таблиці бронювань репетицій;
* `raw_sql_test.js` — окремий скрипт для демонстрації прямих SQL-запитів без використання ORM;
* `.env` — файл із конфіденційними змінними оточення (паролі, порти).

### 4.2 Підключення до MySQL через Sequelize
Ініціалізацію підключення до бази даних винесено в окремий конфігураційний файл `config/database.js`. Дані для авторизації в MySQL безпечно підтягуються з файлу `.env`.

```bash
const { Sequelize } = require('sequelize');
require('dotenv').config();

const sequelize = new Sequelize(
    process.env.DB_NAME,
    process.env.DB_USER,
    process.env.DB_PASSWORD,
    {
        host: process.env.DB_HOST,
        dialect: 'mysql',
        logging: false
    }
);

module.exports = sequelize;
```

### 4.3 Конфігурація Express та запуск сервера
У файлі `server.js` створюється сервер, підключається middleware для парсингу JSON та налаштовується політика CORS. Для лабораторної роботи дозволено запити з будь-якого джерела, щоб frontend, запущений на локальному сервері IDE (наприклад, `localhost:63342`), міг вільно спілкуватися з backend-ом на `localhost:3000`.

```bash
const express = require('express');
const cors = require('cors');
const sequelize = require('./config/database');

const app = express();
app.use(cors());
app.use(express.json());

// ... (Тут підключаються моделі та маршрути) ...

// Синхронізація з БД та запуск сервера
sequelize.sync({ alter: true }).then(() => {
    console.log("✅ З'єднання встановлено, таблиці створено/оновлено успішно!");
    app.listen(3000, () => {
        console.log('🚀 Сервер запущено на http://localhost:3000');
    });
});
```

### 4.4 CRUD-маршрути
Для сутності `Booking` (Бронювання) реалізовано повний набір основних операцій для взаємодії з базою даних:

* `POST /bookings` — створення нового запису на репетицію (Create);
* `GET /users/:id/bookings` — отримання списку бронювань конкретного користувача (Read);
* `PUT /bookings/:id` — оновлення дати та часу репетиції (Update);
* `DELETE /bookings/:id` — скасування (видалення) бронювання (Delete).

Також реалізовано додатковий маршрут `POST /register` для реєстрації нових користувачів із хешуванням паролів та перевірочний маршрут `GET /health` для моніторингу статусу API.

### 4.5 Автоматична генерація таблиць (Синхронізація)
На відміну від систем, що потребують ручного написання SQL-міграцій, у проєкті використовується метод `sequelize.sync({ alter: true })`. Під час кожного запуску сервера Sequelize автоматично аналізує структуру моделей `User` та `Booking`, порівнює їх з поточним станом бази даних у MySQL і динамічно створює або модифікує необхідні таблиці та зовнішні ключі.

## 5. Прямі SQL-запити з Node.js

### 5.1 Підключення через пакет `mysql2`
Для окремої демонстрації ручної роботи з SQL, відповідно до вимог лабораторної роботи, використано пакет `mysql2` та створено скрипт `raw_sql_test.js`.

```bash
require('dotenv').config();
const mysql = require('mysql2/promise');

const connection = await mysql.createConnection({
    host: process.env.DB_HOST,
    user: process.env.DB_USER,
    password: process.env.DB_PASSWORD,
    database: process.env.DB_NAME
});
```

### 5.2 Демонстрація виконання запитів без ORM
У скрипті виконується прямий SQL-запит до створеної бази даних для отримання списку зареєстрованих користувачів. Цей підхід демонструє здатність сервера працювати на низькому рівні з SQL-синтаксисом без використання прошарку абстракцій Sequelize.

```bash
// Виконання сирого SQL-запиту SELECT (Пункт 5 лабораторної роботи)
console.log("⏳ Виконання SQL-запиту: SELECT id, name, email FROM Users...");

const [rows, fields] = await connection.execute('SELECT id, name, email FROM Users');

// Вивід результату у консоль
console.log("\n📋 Результат (Список користувачів):");
console.table(rows);

await connection.end();
```

Цей скрипт запускається незалежно від основного сервера і виводить результати безпосередньо в термінал, доводячи коректність підключення та структуру збережених даних.

## 6. Інтеграція з web-інтерфейсом

### 6.1 Головна сторінка та форми (Створення записів)
Головна сторінка проєкту (`index.html`) містить каталог репетиційних кімнат та інтерфейс для реєстрації/авторизації користувачів. При натисканні на кнопку **"Забронювати"** відкривається модальне вікно. Після заповнення форми клієнтська частина (через `Fetch API`) відправляє `POST`-запит на маршрут `/bookings` (або `/register` для нових користувачів). Дані зберігаються в базу без перезавантаження сторінки.

### 6.2 Особистий кабінет користувача (Відображення даних)
На сторінці кабінету (`cabinet.html`) реалізовано автоматичне завантаження даних користувача. При відкритті сторінки клієнт виконує запит `GET /users/:id/bookings`. Отриманий від сервера JSON-масив бронювань динамічно рендериться у двох форматах:
1. У вигляді зручної таблиці з деталями записів.
2. У вигляді інтерактивного календаря (через бібліотеку FullCalendar) для візуального контролю завантаженості.

### 6.3 Керування бронюваннями (Редагування та Скасування)
Сторінка кабінету виступає повноцінним інтерфейсом для перевірки операцій UPDATE та DELETE без використання сторонніх інструментів (на кшталт Postman). 
* Біля кожного запису в таблиці є кнопка **"Змінити час"**, яка відкриває форму перенесення репетиції (відправляє `PUT`-запит на `/bookings/:id`).
* Кнопка **"Скасувати"** ініціює `DELETE`-запит на сервер, миттєво видаляючи запис із бази даних MySQL та оновлюючи інтерфейс користувача.

### 6.4 Взаємодія через Fetch API
Усі комунікації між клієнтом (браузером) та сервером (Node.js) відбуваються асинхронно за допомогою сучасного стандарту `Fetch API`. У коді (`main.js` та скриптах у `cabinet.html`) реалізовано обробку помилок: якщо сервер вимкнено або запит некоректний, користувач отримує відповідне повідомлення через `alert`, що гарантує стабільність web-інтерфейсу.

## 7. Команди для запуску

Оскільки проєкт не використовує складних монорепозиторіїв та контейнеризації, розгортання середовища та запуск відбуваються максимально прозоро через стандартний менеджер пакетів `npm` та середовище Node.js.

### 7.1 Встановлення залежностей
У кореневій папці бекенду необхідно встановити всі необхідні модулі (Express, Sequelize, MySQL2, CORS, dotenv):
```bash
npm install
```

### 7.2 Підготовка бази даних
1. У MySQL (наприклад, через MySQL Workbench або XAMPP) створити порожню базу даних: `CREATE DATABASE drumspace_db;`
2. Створити у корені проєкту файл `.env` та вказати там параметри доступу (DB_HOST, DB_USER, DB_PASSWORD, DB_NAME).
*(Примітка: Таблиці генерувати вручну не потрібно, Sequelize створить їх автоматично під час першого запуску сервера).*

### 7.3 Запуск серверної частини (Backend / API)
Команда ініціалізує підключення до MySQL, синхронізує моделі з таблицями та запускає HTTP-сервер на порту 3000:
```bash
node server.js
```

### 7.4 Запуск клієнтської частини (Frontend / Web)
Оскільки фронтенд реалізовано на базових веб-технологіях, його достатньо запустити через локальний Live Server у середовищі розробки (наприклад, вбудований сервер PyCharm або розширення Live Server у VS Code). Це дозволить обійти обмеження політики CORS під час тестування:
1. Відкрити файл `index.html` у редакторі.
2. Натиснути іконку браузера для запуску на локальному порту (напр. `http://localhost:63342/.../index.html`).

### 7.5 Запуск демонстрації прямих SQL-запитів
Для перевірки виконання запитів через пакет `mysql2` (без використання ORM) в окремому вікні термінала потрібно запустити підготовлений скрипт:
```bash
node raw_sql_test.js
```

## 9. Висновки

У межах лабораторної роботи для проєкту сервісу бронювання репетиційних баз (DrumSpace) було побудовано реляційну базу даних MySQL і підключено її до серверної частини на платформі Node.js.

Структуру даних (сутності "Користувач" та "Бронювання" зі зв'язком One-to-Many) описано через ORM Sequelize, що дозволило автоматизувати створення таблиць. Реалізовано RESTful API маршрути, які забезпечують виконання повного набору базових CRUD-операцій (створення, читання, оновлення, видалення).

Окремо підготовлено скрипт для прямого виконання SQL-запитів через пакет `mysql2`, щоб продемонструвати розуміння роботи з базами даних на низькому рівні без використання ORM-прошарку.

Додатково backend успішно інтегровано з адаптивним web-інтерфейсом (HTML/CSS/JS). Завдяки використанню Fetch API, операції реєстрації, бронювання кімнат, перенесення часу репетицій та їх скасування працюють асинхронно, без перезавантаження сторінок.

Отримана реалізація повністю відповідає вимогам завдання та є надійною робочою основою (MVP) повноцінного Full-Stack веб-застосунку.
