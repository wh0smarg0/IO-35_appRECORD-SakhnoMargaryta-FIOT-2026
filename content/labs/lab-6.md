## 2. Короткі теоретичні відомості

### 2.1 REST API та HTTP методи
REST (Representational State Transfer) — це архітектурний стиль побудови вебсервісів. Взаємодія базується на стандартних HTTP-методах: `GET` (отримання), 
`POST` (створення), `PUT` (повне оновлення), `PATCH` (часткове оновлення) та `DELETE` (видалення).

### 2.2 Swagger та OpenAPI Specification
Swagger — це набір інструментів для документування та тестування REST API. OpenAPI Specification (OAS) є відкритим стандартом для опису структури API, 
що включає маршрути, методи, параметри та схеми JSON. 


### 2.3 Swagger UI
Swagger UI — це інтерактивний вебінтерфейс, який автоматично генерується на основі специфікації [cite: 186-187]. 
Він дозволяє розробникам переглядати всі доступні ендпоінти та надсилати запити безпосередньо з браузера, що замінює використання 
сторонніх інструментів на кшталт Postman.

### 2.4 Деплой та хмарні платформи
Деплой — це процес публікації готового коду на сервері або хмарній платформі для загального доступу через Інтернет. 
Для Node.js застосунків популярними платформами є Render, Railway та Heroku.

## 4. Реалізація backend-частини (Завдання 1-4)

### 4.1 Підключення MySQL та CRUD-операції
Для підсумкового проєкту реалізовано повноцінне REST API з використанням MySQL та Sequelize ORM. Система підтримує створення користувачів, 
управління репетиційними кімнатами та бронювання часу.

### 4.2 Інтеграція Swagger UI
Встановлено бібліотеки `swagger-ui-express` та `swagger-jsdoc`. У файлі `server.js` налаштовано конфігурацію, яка зчитує JSDoc-коментарі прямо з коду.

```javascript
const swaggerOptions = {
    definition: {
        openapi: '3.0.0',
        info: {
            title: 'DrumSpace API',
            version: '1.0.0',
            description: 'Документація REST API для застосунку DrumSpace',
        },
        servers: [{ url: 'http://localhost:3000' }]
    },
    apis: ['./server.js'], // Шлях до файлу з документацією
};
const swaggerSpec = swaggerJsdoc(swaggerOptions);
app.use('/api-docs', swaggerUi.serve, swaggerUi.setup(swaggerSpec));
```

### 4.3 Документування ендпоінтів
Кожен маршрут задокументовано за допомогою OpenAPI-специфікації. Наприклад, для маршруту отримання кімнат описано схему масиву об'єктів.

```javascript
/**
 * @swagger
 * /api/rooms:
 * get:
 * summary: Отримати список усіх репетиційних кімнат
 * tags: [Rooms]
 * responses:
 * 200:
 * description: Список кімнат успішно отримано
 */
```

Для коректної роботи застосунку на зовнішніх серверах було внесено такі зміни:

3.  **Змінні середовища**: Конфіденційні дані (ключі БД, секрети JWT) винесено в змінні оточення.

## 6. Результати виконання

### 6.1 Робота Swagger UI (Завдання 3, 5)
За адресою `http://localhost:3000/api-docs` доступна інтерактивна документація. 
Усі методи (GET, POST, PUT, DELETE) успішно відображаються у відповідних категоріях (Rooms, Auth, Bookings).

### 6.2 Тестування ендпоінтів
Проведено успішне тестування через функцію **"Try it out"**.
* **Запит `GET /api/rooms`**: Повертає статус `200 OK` та масив кімнат із бази даних MySQL.
* **Запит `/status`**: Демонструє актуальні метрики сервера (uptime, memory usage).

### 6.3 Оптимізація та Кешування
Логи сервера підтверджують, що документація Swagger також користується системою кешування. 
Повторні запити обробляються за **3-5 мс**, що доводить високу продуктивність підсумкового проєкту.
