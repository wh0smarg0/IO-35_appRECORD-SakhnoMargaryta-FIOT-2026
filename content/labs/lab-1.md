## 2 Аналіз предметної області

### 2.1 Актуальність  теми

Актуальність розробки платформи «DrumSpace» зумовлена специфікою гри на ударних інструментах, що потребує спеціально обладнаних приміщень із високим рівнем звукоізоляції. Створення єдиного вебзастосунку дозволить музикантам ефективно знаходити репетиційні бази, порівнювати обладнання та бронювати час у режимі реального часу, що розв'язує проблему відсутності централізованого сервісу в цій галузі.

### 2.2 Об'єкт і предмет

*Об’єкт роботи:* Процес онлайн-бронювання репетиційних майданчиків для музикантів.

*Предмет роботи:* Методи та засоби розроблення адаптивного інтерфейсу та Backend-частини для системи оренди музичного обладнання.

### 2.3 Практичне значення

Результатом роботи є прототип вебзастосунку з адаптивним інтерфейсом та сформованою моделлю предметної області. Отриманий результат може використовуватися як основа для подальшої реалізації backend-частини та розширення функціональності.

### 2.4 Завдання роботи

1.	Проаналізувати предметну область та сформулювати вимоги до системи.
2.	Побудувати Use-case та ER-діаграми для моделювання бізнес-процесів та структури даних.
3.	Розробити адаптивний інтерфейс за допомогою семантичної верстки, Flexbox/Grid та медіа-запитів.
4.	Реалізувати серверну частину на Node.js для обробки запитів до бази даних баз.

## 3. Моделювання системи

### 3.1 Актори системи

* *Музикант (User)* — переглядає каталог баз, використовує фільтри пошуку, переглядає деталі кімнат, здійснює бронювання часу.
* *Адміністратор (Admin)* — керує інформацією про репетиційну базу (додає/редагує кімнати, опис, обладнання), налаштовує розклад та ціни.
* *Система сповіщень* — автоматично надсилає підтвердження на пошту (план на наступні етапи).

### 3.2 Бізнес-логіка
Бізнес-логіка описує правила прийняття рішень та порядок обробки інформації .

1. Користувач може переглядати список баз та розклад вільних годин без реєстрації.
2. Для здійснення бронювання часу користувач повинен бути авторизований у системі.
3. Адміністратор створює прив'язку кімнат до конкретної бази та вказує тип ударної установки.
4. Система перевіряє доступність обраного часу; забронювати один і той самий слот двічі неможливо.
5. Після успішного бронювання статус часу змінюється на зайнятий, а користувачу формується підтвердження.
6. Скасування бронювання можливе не пізніше ніж за 24 години до його початку (умови бізнес-правил).

### 3.3 Функціональні вимоги (FR)
Функціональні вимоги описують, що саме повинна робити система та які функції надавати користувачам.

| --- | --- | --- | --- |
| FR-01 | Перегляд каталогу | Відображення списку репетиційних баз та їх кімнат. | Музикант, Admin |
| FR-02 | Пошук та фільтрація | Фільтрація баз за типом обладнання (акустика/електронні) або розташуванням. | Музикант |
| FR-03 | Перегляд деталей | Відображення детальної сторінки бази з фотографіями та розкладом. | Музикант, Admin |

## 4. Реалізація адаптивного інтерфейсу

### 4.1 Загальна структура сторінок
Інтерфейс побудовано за допомогою семантичних тегів HTML5 (`<header>`, `<main>`, `<section>`, `<article>`, `<footer>`). Для забезпечення гнучкості та адаптивності використано підхід Responsive Web Design на базі чистого CSS (змінні Root, Flexbox, Grid Layout) та Vanilla JavaScript для інтерактивних елементів.

### 4.2 Фрагмент коду: адаптивне бургер-меню (Header)
На десктопних екранах навігація відображається у вигляді горизонтального списку. Для екранів шириною менше `768px` застосовується медіа-запит: стандартне меню приховується (`display: none`), і з'являється іконка `.burger`. 

**CSS-реалізація адаптивності:**
```css
@media screen and (max-width: 768px) {
    .burger { display: flex; }
    .nav-links {
        display: none; /* Приховуємо стандартне меню */
        position: absolute;
        top: 100%; right: 0;
        width: 250px;
        flex-direction: column;
    }
    .nav-links.active { display: flex; } /* Клас додається через JS */
}
````

**JavaScript-логіка перемикання:**
За відображення меню відповідає обробник подій, який додає або забирає клас `.active` при кліку на іконку:

```javascript
const burger = document.getElementById('burger-menu');
const navLinks = document.getElementById('nav-links');

if (burger && navLinks) {
    burger.addEventListener('click', () => {
        navLinks.classList.toggle('active');
    });
}
```

### 4.3 Фрагмент коду: адаптивна сітка каталогу баз (CSS Grid)

Для каталогу репетиційних кімнат використано сучасний підхід до побудови сіток — `CSS Grid` з функціями `repeat`, `auto-fit` та `minmax`. Це дозволяє карткам автоматично підлаштовуватися під ширину екрана без необхідності прописувати додаткові медіа-запити:

```css
.bases-grid {
    display: grid;
    grid-template-columns: repeat(auto-fit, minmax(300px, 1fr));
    gap: 30px;
}
```

*Цей код гарантує, що на мобільних телефонах картки займатимуть 100% ширини вікна, а на десктопах шикуватимуться у декілька колонок (від 300px кожна).*

### 4.4 Сторінки-заглушки для навігації (Модальні вікна)

Щоб уникнути переходів на неіснуючі сторінки (помилка 404) під час розробки прототипу, ключові дії ("Забронювати", "Увійти") реалізовано у вигляді модальних вікон-заглушок. Вони адаптивно підлаштовуються під розмір екрана (`width: 90%; max-width: 450px`).

**HTML-структура модального вікна авторизації:**

```html
<div class="modal" id="login-modal">
    <div class="modal-content stub-content">
        <span class="close-modal">&times;</span>
        <h3>Особистий кабінет</h3>
        <p class="stub-text">Вхід у систему (Заглушка).</p>
        <p class="stub-subtext">Ця сторінка в розробці. У наступній лабораторній роботі тут буде реалізована форма авторизації...</p>
        <div class="stub-buttons">
            <button class="btn btn-primary close-stub">Зрозуміло</button>
        </div>
    </div>
</div>
```
